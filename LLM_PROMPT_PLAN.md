# CSC3170 LLM Integration Plan (Prompt-First)

## Goal
Build an LLM layer for database usage, with primary focus on the bonus requirement:
"Investigate methods for crafting prompts that guide LLM to accurately generate queries for extracting information from the database (5%)."

This project phase prioritizes prompt methodology, evidence logging, and reproducible evaluation.

## Scope
- In scope:
  - NL2SQL backend API for read-only database querying
  - Recommendation API from natural-language preferences
  - Prompt strategy experiments and iteration logs
  - Safety guardrails for generated SQL
- Out of scope in this phase:
  - Full frontend chat page implementation
  - Any write/update/delete SQL generated by LLM

## Architecture Decisions
1. Runtime backend uses Flask + SQLite (movies.db) as primary execution path.
2. LLM model provider starts with Gemini API, wrapped behind a service layer.
3. All generated SQL must pass a safety validator before execution.
4. Prompt experiments are evaluated with a fixed test set and logged.

## Deliverables
1. `POST /api/query/nl`
   - Input: natural language query
   - Output: generated SQL, result rows, execution metadata
2. `POST /api/recommend/nl`
   - Input: natural language preference
   - Output: recommendation list and generated SQL
3. Prompt worklog with versioned experiments
4. Evaluation set and summary metrics

## Workstream A: Prompt Engineering (Core)
1. Build evaluation set:
   - basic lookup
   - filtering
   - aggregation
   - joins
   - sorting/pagination
   - ambiguous and adversarial prompts
2. Compare prompt strategies:
   - zero-shot
   - few-shot
   - constrained template (schema + strict SQL policy)
3. Track each round:
   - prompt version
   - input question
   - generated SQL
   - execution success/failure
   - result correctness
   - issue type and fix
4. Choose final strategy by:
   - SQL executable rate
   - answer correctness rate
   - latency
   - robustness under invalid input

## Workstream B: Backend Implementation
1. Create LLM service module:
   - prompt builder
   - Gemini call
   - SQL extraction
   - recommendation helper
2. Add SQL safety gateway:
   - allow SELECT-only semantics
   - block dangerous keywords
   - single-statement only
   - enforce LIMIT
3. Add Flask routes:
   - `/api/query/nl`
   - `/api/recommend/nl`
4. Add consistent error format for model failures and SQL validation failures

## Verification
1. Run evaluation set through each prompt strategy.
2. Report:
   - executable SQL rate
   - correct-answer rate
   - average latency
   - failure distribution
3. Run adversarial tests:
   - injection-like prompt
   - broad query without limit
   - unsupported operation prompt
4. Confirm output schema is stable for frontend integration.

## Collaboration Contract (for frontend teammate)
Backend returns a stable JSON format with:
- `query`
- `generated_sql`
- `results`
- `result_count`
- `latency_ms`
- `errors` (if any)

This allows frontend integration later with minimal rework.
